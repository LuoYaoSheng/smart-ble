## 一、入门篇

### Q1: BLE 和经典蓝牙有什么区别？

| 特性 | 经典蓝牙 | BLE（低功耗蓝牙） |
|------|---------|------------------|
| 功耗 | 高 | 极低 |
| 连接速度 | 慢 | 快 |
| 传输速度 | 快 | 慢 |
| 连接数 | 7 个 | 理论无限 |
| 应用场景 | 音频、数据传输 | 传感器、控制 |

**一句话**：BLE 省电但速度慢，适合传感器和控制类应用。

---

### Q2: UUID 是什么？怎么选？

UUID（通用唯一标识符）用于标识服务和特征值。

**类型**：

| 类型 | 长度 | 示例 | 用途 |
|------|------|------|------|
| 16-bit | 4 字符 | 0x180A | 标准服务 |
| 32-bit | 8 字符 | 0x12345678 | 较少用 |
| 128-bit | 32 字符 | 4FAFC201-... | 自定义服务 |

**推荐**：自定义服务使用 128-bit UUID，避免冲突。

```bash
# 在线生成 UUID
https://www.uuidgenerator.net/
```

---

### Q3: 服务和特征值是什么关系？

```
设备
└── 服务 (Service)
    └── 特征值 (Characteristic)
        ├── 值 (Value)
        └── 描述 (Descriptor)
```

**比喻**：
- 设备 = 一个公司
- 服务 = 部门（销售部、技术部）
- 特征值 = 具体功能（查报价、提工单）

---

### Q4: 读写通知有什么区别？

| 操作 | 方向 | 用途 |
|------|------|------|
| **Read** | 设备 → 手机 | 读取设备状态 |
| **Write** | 手机 → 设备 | 发送控制命令 |
| **Notify** | 设备 → 手机 | 主动推送数据 |
| **Indicate** | 设备 → 手机 | 带确认的推送 |

**建议**：数据更新频繁用 Notify，需要确认用 Indicate。

## 二、开发篇

### Q5: 为什么微信小程序 BLE API 和原生不一样？

微信小程序运行在沙箱环境，权限受限：

```javascript
// 微信小程序
wx.startBluetoothDevicesDiscovery({
  success: () => {}
})

// Android 原生
bluetoothAdapter.startLeScan(scanCallback)

// 完全不同！
```

**解决**：使用 Smart BLE 的抽象层，统一 API。

---

### Q6: Android 12+ 蓝牙扫描不到设备？

Android 12 开始需要新权限：

```xml
<!-- AndroidManifest.xml -->
<uses-permission android:name="android.permission.BLUETOOTH_SCAN" />
<uses-permission android:name="android.permission.BLUETOOTH_CONNECT" />
```

还需要运行时请求权限：

```kotlin
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES_CODES.S) {
    requestPermissions(arrayOf(
        Manifest.permission.BLUETOOTH_SCAN,
        Manifest.permission.BLUETOOTH_CONNECT
    ), 100)
}
```

---

### Q7: iOS 蓝牙扫描必须要有定位权限？

是的，iOS 认为 BLE 扫描可能获取位置信息：

```xml
<!-- Info.plist -->
<key>NSBluetoothAlwaysUsageDescription</key>
<string>需要蓝牙权限连接设备</string>

<key>NSBluetoothPeripheralUsageDescription</key>
<string>需要蓝牙权限</string>

<key>NSLocationWhenInUseUsageDescription</key>
<string>需要定位权限扫描蓝牙设备</string>
```

### Q8: 连接后为什么自动断开？

**常见原因**：

1. **连接参数不当**
   ```
   Android 默认: 7.5ms - 30ms
   iOS 默认: 15ms - 30ms
   ESP32 默认: 30ms - 50ms
   ```

   解决：调整连接间隔

2. **超时断开**
   - 部分设备有超时机制（30秒无通信断开）
   - 解决：定期发送心跳包

3. **信号太弱**
   - RSSI < -90 dBm 容易断开
   - 解决：靠近设备

---

### Q9: 如何处理 BLE 数据格式？

BLE 传输的是字节数组，常用格式：

**HEX 格式**：`FF 01 AB CD`
- 适合：二进制协议
- 显示：紧凑但可读性差

**UTF-8 格式**：`{"cmd":"on"}`
- 适合：文本协议、JSON
- 显示：可读性好

**Base64**：`SGVsbG8=`
- 适合：传输二进制数据

```typescript
// HEX 转 ArrayBuffer
function hexToArrayBuffer(hex: string): ArrayBuffer {
  const bytes = hex.match(/[\da-f]{2}/gi) || []
  return new Uint8Array(bytes.map(b => parseInt(b, 16))).buffer
}

// ArrayBuffer 转 HEX
function arrayBufferToHex(buffer: ArrayBuffer): string {
  return Array.from(new Uint8Array(buffer))
    .map(b => b.toString(16).padStart(2, '0'))
    .join(' ')
}
```

## 三、调试篇

### Q10: 扫描不到设备怎么办？

**排查清单**：

```
□ 设备是否开启广播
□ 手机蓝牙是否打开
□ 距离是否太远（< 10米）
□ 设备是否被其他设备连接
□ Android 是否授予位置权限
□ iOS 是否授予蓝牙+定位权限
□ 设备名称过滤是否过于严格
□ RSSI 过滤是否过于严格
```

**工具**：
- Android: nRF Connect
- iOS: LightBlue / nRF Connect
- 桌面: Smart BLE Desktop

---

### Q11: 连接超时怎么办？

**原因**：

1. **设备已连接**
   - BLE 设备通常只能被一个设备连接
   - 先断开其他连接

2. **连接参数不匹配**
   ```javascript
   // 尝试不同的连接参数
   await device.connect({
     timeout: 10000,  // 超时时间
     autoConnect: false
   })
   ```

3. **设备进入休眠**
   - 部分设备会自动休眠
   - 先发送数据唤醒

---

### Q12: 写入失败 "write failed"？

**检查点**：

1. **特征值属性**
   ```
   确认特征值支持 WRITE 或 WRITE_NO_RESPONSE
   ```

2. **数据长度**
   ```
   单次写入最大 20 字节（MTU 限制）
   超过需要分包
   ```

3. **数据格式**
   ```
   确认设备期望的格式（HEX/UTF-8）
   ```

4. **需要先订阅**
   ```
   某些设备要求先订阅才能写入
   ```

---

### Q13: 订阅后收不到通知？

**排查**：

1. **确认特征值支持 Notify**
   ```
   检查 properties 是否包含 notify
   ```

2. **确认设备主动发送**
   ```
   通知是设备主动推送的
   某些设备需要先写入命令才会通知
   ```

3. **检查 CCC 描述符**
   ```javascript
   // 某些平台需要手动写 CCC
   await descriptor.writeValue([0x01, 0x00])
   ```

---

### Q14: MTU 是什么？重要吗？

MTU（Maximum Transmission Unit）= 最大传输单元

| MTU | 单次传输 | 传输 100 字节 |
|-----|---------|-------------|
| 23 (默认) | 20 字节 | 5 次 |
| 185 | 182 字节 | 1 次 |

**建议**：连接后请求更大的 MTU：

```javascript
// 请求 MTU=185
await device.requestMtu(185)
```

## 四、硬件篇

### Q15: ESP32 和 ESP32-C3 怎么选？

| 型号 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| ESP32 | 功能全、IO 多 | 体积大、功耗稍高 | 复杂项目 |
| ESP32-C3 | 便宜、小巧 | IO 少 | 简单传感器 |

**BLE 两者一样**，选 ESP32-C3 更划算。

---

### Q16: ESP32 广播间隔设置多少合适？

```
推荐值：
- 快速发现：30-50ms（耗电）
- 平衡模式：100-200ms
- 省电模式：500-1000ms（发现慢）
```

```cpp
// 设置广播间隔
esp_ble_adv_params_t adv_params = {
    .adv_int_min = 160,  // 100ms (N*0.625ms)
    .adv_int_max = 320,  // 200ms
    // ...
};
```

---

### Q17: ESP32 连接数有限制吗？

**默认**：最多 3 个同时连接

**修改**（需要足够的内存）：

```cpp
#define CONFIG_BT_ACL_CONNECTIONS 9  // 最多9个
```

**注意**：连接越多，内存和CPU占用越高。

### Q19: 如何保证数据安全？

**措施**：

1. **加密传输**
   ```
   使用 LE Secure Connections
   ```

2. **设备认证**
   ```
   绑定 MAC 地址
   使用配对码
   ```

3. **数据校验**
   ```
   CRC 校验
   消息序列号
   ```

---

### Q20: 如何测试 BLE 性能？

**指标**：

| 指标 | 测量方法 | 目标 |
|------|---------|------|
| 连接时间 | 发起连接到连接成功 | < 3s |
| 扫描延迟 | 开始扫描到发现设备 | < 2s |
| 吞吐量 | 每秒传输字节数 | 视场景 |
| 功耗 | 测量电流 | < 10mA |

**工具**：
- 功耗分析仪
- Nordic nRF Sniffer
- Smart BLE 自带日志
