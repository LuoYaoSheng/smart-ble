# 跨平台蓝牙日志抓取与沉淀规范 (Logging Specification)

蓝牙应用在现场部署时往往面临极度恶劣的信号环境与硬件碎片化问题。为了确保开发者能第一时间通过日志复盘并溯源崩溃节点，所有的 Smart BLE 前端 SDK（Flutter, UniApp, Tauri, Native）必须强制遵循下述结构化的日志规范（Logging Spec）。

---

## 1. 六级状态池机制 (6-Level Typed Pool)

所有的底层分发必须包含严格的 6 种类型判定。禁止随意使用万能的 `.log()` 输出废话，每条日志必须包含时间戳和准确的标签标识。

| 级别分类 | 标识色 / UI 表征 | 适用场景 |
| :--- | :--- | :--- |
| `INFO` | ⚪ 白色 / 灰色 | 一般事件节点。如：“系统已就绪”、“发现服务 180A” |
| `SUCCESS` | 🟢 绿色 | 极其关键的里程碑宣告。如：“十六进制 OTA 包切片发送完毕”、“设备配对成功” |
| `WARNING` | 🟡 橙色 | 不引发崩溃但非预期的现象。如：“CRC8 校验错误，抛弃错帧”、“信道断流重传” |
| `ERROR` | 🔴 红色 | 严重错误、中断。如：“连接看门狗超时断开”、“设备拒收命令” |
| `SEND` | 🔵 蓝色 (带发送图标) | **上行通道**：向 `Write`/`WriteNoResponse` 特征写入的数据流。必须含有对应的 HEX |
| `RECEIVE`| 🟣 紫色 (带接收图标) | **下行通道**：从 `Notify`/`Indicate`/`Read` 收到的硬件数据流。必须含有对应的 HEX |

## 2. 内存环形调度 (Ring-Buffer Sandboxing)

由于移动端设备的资源极为受限，尤其是后台静默记录低功耗蓝牙长时间的心电图/温湿度曲线时极易造成 Out of Memory (OOM)。

**强制规约：**全端组件必须应用最大 **1,000 ~ 5,000 条**（取决于目标操作系统的可用基础 RAM）的循环队列覆盖写入模型。当内存超载越界时，剔除最早期记录，永不导致设备瘫痪。

## 3. 持久化与排障导出协议 (Export Protocol)

仅仅依赖 `console.log` 的跨端应用将完全失去真实场景联调的机会。我们的核心定位是商业级交付器，系统必须具有跨维度的反查手段：

### (1). 导出内容编排
导出的日志并不是一堆纯粹的文本，为了方便 `LogStash` 或是其它自研工具查阅比对，导出协议的格式强烈建议兼容 **JSON-Lines (NDJSON)** 并附带易读的纯文本前缀元数据。

```json
{ "timestamp": "2026-04-14T03:32:01.002Z", "type": "SEND", "device": "F1:A2:33", "hex": "BE B5 01 02", "msg": "Send command chunk" }
```

### (2). 平台调用矩阵
禁止使用单一的“复制到剪贴板”来糊弄普通用户（文本往往极大，剪贴板装载失败率极高）。

* **移动端 (Flutter / UniApp)**
  - 必须利用原生调起 `ShareFiles` 动作！生成 `.log` 或 `.txt` 文件直接调用系统分享弹窗。开发者或用户可借此一键飞书 / 微信 / 邮件中转日志文本。
* **桌面跨端 (Electron / Tauri)**
  - 必须唤出操作系统的 **Save Dialog 资源框**，强制以二进制或 UTF-8 流存入硬盘，方便进行长时间的大规模记录挂机测试。
