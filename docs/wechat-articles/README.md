# Smart BLE 公众号系列规划

> 用途：规划 `smart-ble` 的公众号与长文内容线
> 角色：既服务项目传播，也服务教程体系和多分支演进记录

---

## 这组内容的定位

`smart-ble` 不是普通单应用项目，所以公众号内容不能只写成“产品宣传”。

这组内容要同时承担 3 个任务：

1. **项目入口**：让新读者快速知道 Smart BLE 是什么
2. **学习路径**：让开发者知道从哪里开始学 BLE、多平台实现和硬件联动
3. **演进记录**：解释为什么它会保留多实现、多分支和历史资产

---

## 内容主线

建议把公众号内容拆成 4 条主线：

### 1. 项目认知线

适合第一次接触项目的读者。

典型内容：

- 项目介绍
- 为什么做 Smart BLE
- 为什么不是单一实现
- 如何选择适合自己的版本

### 2. 教学实战线

适合想学习 BLE 和跨平台实现的读者。

典型内容：

- Flutter BLE 实战
- uni-app BLE 能力
- CoreBluetooth / Android BLE
- ESP32 硬件联动
- 外设模式 / OTA / 协议设计

### 3. 平台对比线

适合想做技术选型和理解多实现差异的读者。

典型内容：

- 8 种实现如何分工
- Tauri / Electron / 原生 / Flutter 的取舍
- 小程序与正式 App 的区别

### 4. 更新与路线图线

适合持续关注项目的人。

典型内容：

- 开发规划
- 版本更新
- Bug 修复记录
- 阶段总结

## 写作原则

### 原则 1：不要只写“我们做了什么”

还要写：

- 为什么这样做
- 为什么保留多个实现
- 不同实现分别适合谁

### 原则 2：教学内容优先保留路径感

一篇文章最好明确：

- 适合谁读
- 前置知识是什么
- 读完应该去看哪个版本或哪个目录

### 原则 3：历史分支不要当包袱处理

像 `LightBLE`、旧 iOS 资产这类内容，不一定都要高频宣传，但在文章里可以作为：

- 演进背景
- 设计决策来源
- 多实现路线的历史证据
